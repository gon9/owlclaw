## 0. 前提と用語

- **自立稼働**: 毎回プロンプトしなくても、エージェントが「一度言われたこと」を覚えていて、適切なタイミングで自発的に動く状態。
- **自立 ≠ 万能**: エージェントは "自分で考える" のではなく、`Standing Order`（永続指示）と `トリガ`（時間 or イベント）の組合せで動く。
- **owlclaw の現状**: 1 つの Task（daily-digest）を Claude Code CLI 経由で 1 日 1 回走らせる Python+Bash のパイプライン。
- **目指す姿**: Task が複数あり、それぞれ独立した source / prompt / schedule / output を持つ汎用 "AI Task Runner"。

各ユースケースのフォーマット:

- **想定シーン**: 自分視点での体験
- **トリガ**: いつ動くか
- **入力データ**: 何を読むか
- **出力**: 何を返すか
- **失敗 / 嫌な挙動**: 避けたいこと
- **価値**: 作る理由
- **完成判定 (DoD)**: ここまで動けば「できた」と言える線
- **必要な能力**: タスク実装に必要な汎用機能
- **owlclaw 現状とのギャップ**: 今の実装に何が足りないか

## UC1. カレンダー → 出発時刻提案

### 想定シーン
明日 14:00 に表参道で打合せ、と Google カレンダーに入れた瞬間、その日の朝に「13:20 までに家を出れば間に合います。雨予報なので傘」と通知が来る。当日の朝、寝坊しても出発時刻だけは忘れない。

### トリガ
- 予定登録/更新の検知（push、または定期 fetch）
- 当日 7:00（fallback の cron）
- 出発予定時刻の 30 分前（リマインド）

### 入力データ
- Google Calendar 予定（場所、開始時刻、参加者）
- 自宅 / 会社の住所（プロファイル）
- 当日の交通情報・天気（Web API）
- 移動所要時間の概算（学習可能）

### 出力
- 「出発時刻 X:XX」「経路概要」「持ち物（傘 / ID 等）」を 1 メッセージで Slack へ
- リマインド時刻に再通知（簡略版）

### 失敗 / 嫌な挙動
- リモート会議なのに出発時刻を出す → 場所欄が URL/オンラインなら抑制
- 同じ予定について何度も通知 → 1 予定 1 通知の重複抑止
- 早朝/深夜に通知 → アクティブ時間外は抑制

### 価値
- 移動所要時間の見積りを毎回スマホで叩く必要がなくなる
- 「忘れ物・出発遅れ」のリスクを構造的に潰せる

### 完成判定 (DoD)
- 1 週間連続で「物理外出予定」に 100% 自動通知
- リモート予定では通知が出ない
- 1 予定につき通知は最大 2 回（前日夜 + 30 分前）

### 必要な能力
- カレンダー source（Google Calendar API or ICS 取得）
- 場所判定（URL/物理住所の分岐）
- 移動時間推定（Google Maps Directions API or 静的辞書）
- 通知済み予定の永続化（state store）
- one-shot cron（出発時刻 -30 分の動的登録）

### owlclaw 現状とのギャップ
- ❌ Source は RSS のみ → calendar source プラグインを追加する必要
- ❌ State 永続化なし（Obsidian ファイルに痕跡は残るが構造化されない）→ 軽量 state store（JSON or SQLite）を新設
- ❌ one-shot cron なし → mac launchd / cron の動的登録 helper が必要
- ✅ Slack 通知は流用可能（`scripts/slack_notify.sh`）

## UC2. 旅行の予約状況チェック / プラン補完

### 想定シーン
航空券予約メールが届く → 自動で「旅程」として記録 → 出発 14 日前「ホテル未予約」、7 日前「ビザ要件」、3 日前「現地天気・電源プラグ」、1 日前「持ち物リスト」と段階通知。

### トリガ
- 予約メール検知
- 旅程登録後、D-14 / D-7 / D-3 / D-1 / 当日朝（cron 相対）

### 入力データ
- Gmail（航空券・ホテル・レンタカー等のメール）
- カレンダー（旅行の予定ブロック）
- Web 検索（現地天気、ビザ、電源、治安、祝日）
- 過去の旅行ログ（Obsidian の `Travel/` フォルダ）

### 出力
- 旅程台帳: `Obsidian/Travel/2026-05-osaka.md`（自動メンテ）
- フェーズごとのチェックリスト Slack 通知
- 不足項目だけ強調

### 失敗 / 嫌な挙動
- 出張と私用旅行を混ぜる → タイトル/参加者で分離タグ付け
- 同じチェックリストを何度も送る → 段階遷移時のみ通知
- メール解析失敗 → 確信度が低ければ確認質問のみ

### 価値
- 旅行準備の「忘れた」を構造的に防ぐ
- 過去旅行データから経験値が蓄積する

### 完成判定 (DoD)
- メール検知 → 旅程台帳生成までが 1 旅程で完走
- D-14 / D-7 / D-3 / D-1 のチェックインが時系列で出る
- 既予約項目は重複チェックされない

### 必要な能力
- Gmail source
- メール内容パース（旅程抽出 → 構造化 yaml/md）
- Obsidian への構造化書き出し（既存の `write_obsidian.sh` 拡張）
- 多段 cron（D-N 自動登録）
- 旅程ステート（予約済 / 未予約のフラグ管理）

### owlclaw 現状とのギャップ
- ❌ Gmail source なし → 新規プラグイン
- ❌ 構造化 state 管理なし
- ❌ 多段 cron なし
- ✅ Obsidian 書き出しは流用可能（`scripts/write_obsidian.sh`）
- ✅ Claude による解析は既存パイプラインの自然な拡張

## UC3. メール → 月次支払いサマリ・予算警告

### 想定シーン
毎週日曜夜「今週の決済 12 件、合計 ¥48,200。月予算 ¥150k に対し 78% 消化。サブスク NETFLIX の更新が水曜にあります」と通知。月末「外食が前月比 +35%」のような異常検知も。

### トリガ
- 決済メール受信時（即時カウント、サイレント）
- 毎週日曜 21:00（cron でサマリ送信）
- 月予算 50% / 80% / 100% 到達時（しきい値ベース）

### 入力データ
- Gmail 決済メール（カード明細、PayPay、Amazon、Apple、Google Play 等）
- 月予算（プロファイル、カテゴリ別あれば理想）
- 過去 3 ヶ月の支払履歴（異常検知ベースライン）

### 出力
- 週次サマリ（合計、件数、カテゴリ Top 3、サブスク注意）
- しきい値到達時の即時アラート
- 月末レポート（前月比、異常カテゴリ）

### 失敗 / 嫌な挙動
- 同じ取引を二重カウント → 重複排除ロジック必須
- 銀行/カード番号を平文で残さない
- 集計ミスで信頼を失う → 月初に「先月の生サマリ確認」を 1 回挟む

### 価値
- 「気づいたら使いすぎ」を防ぐ
- サブスク棚卸しが半自動化
- 自立稼働の効果が一番数値で見える

### 完成判定 (DoD)
- 4 週連続で日曜 21:00 にサマリが届く
- 80% ラインで自動アラート
- 二重カウント率 < 5%

### 必要な能力
- Gmail source（フィルタ条件をタスクごとに設定可能）
- 構造化 state（月次累計、カテゴリ集計、メール ID 重複排除）
- しきい値判定とエッジトリガ（一度しか送らない仕組み）
- カテゴリ推定（Claude プロンプト or ルール）

### owlclaw 現状とのギャップ
- ❌ Gmail source なし
- ❌ 累計 state なし
- ❌ しきい値エッジ判定なし
- ✅ Claude による集計とサマリ生成は既存パイプラインの型
- ✅ Slack 通知流用可能

## UC4. 誕生月の有益情報

### 想定シーン
誕生月の 1 日朝「今月誕生日。免許更新 / 健康診断対象月 / 自治体特典 / 保険見直しおすすめ / 税制で 12 月までに考えるべき項目」を 1 メッセージで。

### トリガ
- 毎年の誕生月 1 日 8:00（cron, yearly）
- 誕生月中の中旬リマインド（1 回）

### 入力データ
- 誕生日（プロファイル）
- 居住自治体（プロファイル）
- 保険 / 税 / 健康関連の Web 情報（検索）

### 出力
- 1 メッセージで網羅（チェックリスト形式）
- 期限ある項目は別途 cron でフォローアップ作成

### 失敗 / 嫌な挙動
- 一般論しか出ない → ユーザー固有情報をプロンプトに必ず注入
- 毎月送る → 誕生月のみ

### 価値
- 実装工数が一番小さい（cron 1 発）
- 年 1 回でも面倒な棚卸しを 1 回で終える価値あり

### 完成判定 (DoD)
- 誕生月 1 日 8:00 に通知
- 5 項目以上の具体提案
- 「すでに済」とユーザーが返したら次月以降除外

### 必要な能力
- cron（yearly）
- プロファイル機構（誕生日・居住地・年齢を Claude に渡す）
- Web 検索 / fetch（Claude Code CLI の標準ツールで OK）
- 完了履歴の永続化

### owlclaw 現状とのギャップ
- ❌ cron は外部依存（mac launchd or systemd timer）→ schedule 設定例の整備
- ❌ プロファイル機構なし（プロンプトに毎回手で書く構造）→ `profile.yaml` を新設
- ✅ Claude による Web 検索 + 整形は既存パイプラインで実現可能

## UC5. 訪問先（顧客 / オフィス）事前ブリーフィング

### 想定シーン
「来週水曜 14:00 ナレッジワーク本社」とカレンダーに入った瞬間、前日 20:00 に「事業概要 / 直近ニュース 3 本 / 担当者の公開情報 / 過去会話ログ要約 / 質問たたき台 5 件」が届く。

### トリガ
- 外部組織との予定検知時（即時タグ付け）
- 予定の前日 20:00（cron 相対）
- 当日 1 時間前（簡略リマインド）

### 入力データ
- カレンダー予定（参加者ドメイン、件名、場所）
- 会社サイト / プレス / Wantedly / ブログ（Web fetch）
- 過去ログ（Obsidian 内、検索）
- LinkedIn 等の公開情報（要 API or 手動補助）

### 出力
- 1 メッセージのブリーフィング Slack
- `Obsidian/Briefings/2026-05-XX-knowledgework.md` 保存

### 失敗 / 嫌な挙動
- 社内予定にも反応 → 参加者ドメインでフィルタ
- ハルシネーション → 必ずソース URL を併記
- プライベート予定（飲み会等）→ 件名/タグで除外

### 価値
- cron 脱却が一番効く（カレンダー event がそのまま wake する）
- 商談 / 面談の質が上がる

### 完成判定 (DoD)
- 外部予定検知 → 前日 20:00 のブリーフィング送信が 5 件連続成功
- 内部予定では発火しない
- 各情報項目にソース URL が併記

### 必要な能力
- Calendar source（UC1 と共通）
- ドメイン判定ルール
- Web 検索 + fetch
- Obsidian 過去ログ参照（`Obsidian/Daily Notes/` を検索）
- 通知済みフラグ管理

### owlclaw 現状とのギャップ
- ❌ Calendar source（UC1 と共通の足し算）
- ❌ Obsidian 検索ロジック（read 専用 helper を新設）
- ✅ Claude による調査・整形は既存パイプラインの型
- ✅ Obsidian 書き出し流用

## UC6. 「parantia のブログ読んできて」型タスク

### 想定シーン
人が一度「parantia のブログ読んできて」と言う → エージェントが読んでサマリ → 「次から週 1 で新着チェックする?」と聞き返す → Yes なら Standing Order に昇格 → 以降毎週月曜 9:00 に新着サマリ。

### トリガ
- 1 回目: ユーザー直接指示（自律ではない普通のタスク）
- 昇格後: 週次 cron + フィード差分検知

### 入力データ
- 対象 URL（RSS or HTML）
- 過去送信済みの記事 ID（重複防止）

### 出力
- 1 回目: 直近 5 件の要約
- 自律後: 週次で「新着 N 本、要約と気づき」

### 失敗 / 嫌な挙動
- 新着なしの週も律儀に通知 → "新着なし" は無音
- 重複通知 → 永続化された送信履歴を参照

### 価値
- Standing Order 昇格パターンの代表例
- owlclaw のコア機能の自然な拡張（既に RSS digest は動いている）

### 完成判定 (DoD)
- 1 回目の指示で要約が返る
- 昇格提案が必ず最後に提示される
- 昇格後 4 週連続で正しく動く

### 必要な能力
- 既存の RSS fetch 機構の再利用
- 動的 source 追加（`config/sources.yaml` への自動追記、または別ファイル）
- 送信済み記事 ID の永続化（既存パイプにあるか要確認）
- 昇格ダイアログの UX（Slack インタラクションか CLI 対話）

### owlclaw 現状とのギャップ
- ✅ RSS fetch は既存（`scripts/fetch_rss.py`）
- ❌ ソース動的追加 → CLI コマンドか Slack ボット経由で `sources.yaml` を編集する仕組み
- ❌ 過去送信記事の永続化（fetch 時の重複排除のみで、Slack 配信側の重複はチェックしてない可能性）
- ❌ 昇格 UX

| UC | 実装工数 | 体感価値 | 設計学習効果 | owlclaw からの距離 |
|----|---------|---------|-------------|-------------------|

| UC5 ブリーフィング | 中 | 大 | **高** | 中 |
| UC2 旅行 | 大 | 中 | 高 | 遠い |

→ 推奨 MVP 順序: **UC6（既存に最も近い） → UC4（最小工数） → UC3（最大体感）→ UC1/UC5（Calendar 投資の回収） → UC2（応用）**
